Interpretation

The dashboard is generated from a single source file, outputs/compliance_report.json, which contains 33 trades and 66 regime-level decisions across CFTC, EMIR. The portfolio heatmap shows a strongly exception-heavy reporting book: 51 regime outcomes are non-compliant, compared with 7 compliant outcomes, 3 conditional outcomes, and 5 not-applicable outcomes. This matters because the failures are not confined to one isolated trade; non-compliant trades include T001, T002, T003, T004, T005, T006, T007, T008, T009, T010, T011, T012, T013, T014, T015, T016, T017, T018 and others, indicating a portfolio-level control problem rather than a single booking mistake.

The error-frequency chart narrows the operational problem to repeatable data-quality checks. The most frequent message groups are other_counterparty_lei (26), reporting_counterparty_lei (22), LEI (22), collateral_portfolio_code (14). These are exactly the types of fields that a pre-submission validation layer should catch: counterparty LEI format and checksum controls, mandatory collateral and margin fields, UTI consistency, and reference-rate hygiene. In other words, the dashboard does not merely count failed trades; it points to the specific controls that would reduce the greatest number of reporting breaks.

The asset-class breakdown gives useful context for the classification question. The source data covers Rates: 11, FX: 6, Equity: 5, Credit: 4, EventContract: 4, Commodities: 3, so the event-contract trades are being compared against a mixed OTC derivative portfolio rather than viewed in isolation. Conventional asset classes mainly fail because required reporting fields or identifiers are invalid. By contrast, the event-contract records fail to map into an ANNA-DSB UPI product definition, which makes their issue a classification boundary problem rather than an ordinary completeness problem. The classification frontier panel therefore deserves separate treatment: the highlighted trades are T026 (PoliticalOutcome), T027 (MacroeconomicOutcome), T028 (RegulatoryDecisionOutcome), T033 (MonetaryPolicyOutcome), executed on Kalshi, Polymarket. The regime outcome count is CONDITIONAL: 3, NOT_APPLICABLE: 5. Kalshi-style regulated event contracts are treated as conditional under the CFTC logic, while Polymarket/offshore or EMIR-facing event contracts are outside the reporting chain used in this project. The additional test set still includes clean controls (T029, T032) and designed data exceptions (T030, T031, T033), so the dashboard separates three concepts that should not be mixed: valid trades, data reporting failures, and true taxonomy gaps.
